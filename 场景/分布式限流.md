### 一、设计分布式限流工具，首先要考虑的核心维度

在动手设计前，必须先明确这些基础问题，避免方案偏离实际需求：

#### 1. 限流目标

- 粒度：是**集群总 QPS**（如整个服务最多处理 1000QPS）、**单接口集群 QPS**（如订单接口最多 500QPS）、还是**单用户 / IP 集群限流**（如单个用户 10QPS）；
- 规则：是固定阈值（如 1000QPS）、动态阈值（如高峰期降为 800）、还是突发流量允许（如令牌桶的 “桶容量”）；
- 限流动作：是直接拒绝（返回 429）、排队等待（漏桶）、还是降级（返回兜底数据）。

#### 2. 一致性与性能的权衡

- 强一致性：集群限流严格精准（如总 QPS 必须≤1000），但会引入分布式锁 / 网络开销；
- 最终一致性：允许短期小幅超阈值（如瞬间 1050QPS），但性能更高（适合大部分场景）。

#### 3. 高可用与容错

- 限流组件本身不能是单点（如依赖的存储挂了，是否降级为单机限流）；
- 网络延迟 / 超时：获取令牌超时，是拒绝请求还是走兜底策略。

#### 4. 可观测性

- 需监控：集群总流量、已用令牌数、拒绝次数、令牌发放延迟；
- 需可配置：支持动态调整阈值（无需重启服务）。

### 二、分布式限流如何控制集群整体？核心方案

分布式限流的核心是**将 “限流计数器 / 令牌” 从单机收敛到一个全局存储**，所有节点共享同一套限流状态，常见方案分两类：








|        方案         |                           核心思路                           |        优点        |          缺点          |        适用场景        |
| :-----------------: | :----------------------------------------------------------: | :----------------: | :--------------------: | :--------------------: |
|     中心化存储      | 用 Redis/ZooKeeper 存储全局计数器 / 令牌池，所有节点操作该存储 | 一致性强、规则统一 | 依赖中间件、有网络开销 | 集群规模大、需精准限流 |
| 本地限流 + 全局协调 | 每个节点预分配限流额度（如总 1000QPS，10 个节点各 100），定期校准全局额度 |   性能高、无单点   |  节点扩缩容时额度不均  | 节点数稳定、允许小幅超 |
| 分布式令牌桶 / 漏桶 | 将令牌桶的 “令牌生成与存储” 放到中心节点，所有节点从中心获取令牌 | 兼容单机令牌桶逻辑 |     中心节点压力大     |   需平滑处理突发流量   |

#### 核心方案落地：Redis 实现集群限流（最常用）

Redis 天生适合做分布式限流，支持原子操作（保证一致性）、高性能（支撑高并发），主要有两种方式：

##### 1. 计数器法（简单集群限流）

适合控制 “单位时间内总请求数”（如 1 秒 1000QPS），利用 Redis 的`INCR`+`EXPIRE`原子操作：










```
// 伪代码：Redis实现集群QPS限流
public boolean allowRequest(String key, int limit, int expireSeconds) {
    // 1. 原子自增，key为限流标识（如"order_api:qps"）
    Long count = redisTemplate.opsForValue().increment(key, 1);
    // 2. 首次设置过期时间（避免多个节点重复设置）
    if (count == 1) {
        redisTemplate.expire(key, expireSeconds, TimeUnit.SECONDS);
    }
    // 3. 判断是否超过阈值
    return count <= limit;
}
```

- 问题：可能存在 “时间窗口临界值”（如 00:00:00.999 加了 999，00:00:01.001 又加了 999，1 秒内实际 1998）；
- 优化：用 Redis 的`HINCRBY`按毫秒分桶，或用 Lua 脚本保证 “自增 + 过期” 原子性。

##### 2. 分布式令牌桶（重点解答你的问题）

单机令牌桶的问题是 “令牌只在本地生成，集群无法共享”，分布式改造的核心是**将令牌池放到 Redis，所有节点从 Redis 获取令牌**，步骤如下：

### 三、令牌桶适配分布式场景：令牌的存储与分发

#### 1. 核心改造思路

- 令牌不再由每个节点本地生成，而是由**Redis 统一生成和存储令牌池**；
- 所有服务节点请求令牌时，都向 Redis 申请，Redis 原子性地扣减令牌；
- 令牌生成逻辑：Redis 通过定时任务（或 Lua 脚本）按固定速率补充令牌（如每秒生成 1000 个），令牌池有最大容量（应对突发流量）。

#### 2. 具体实现（Redis + Lua 脚本，保证原子性）

Lua 脚本能将 “检查令牌数→补充令牌→扣减令牌” 的逻辑原子化，避免并发问题，核心脚本如下：











```
-- 分布式令牌桶Lua脚本
-- 参数：KEYS[1] = 令牌桶key，ARGV[1] = 令牌生成速率（每秒生成数），ARGV[2] = 桶最大容量，ARGV[3] = 当前时间戳，ARGV[4] = 申请的令牌数
local key = KEYS[1]
local rate = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local apply = tonumber(ARGV[4])

-- 1. 获取令牌桶当前状态（last_time：上次补充令牌时间，tokens：当前令牌数）
local bucket = redis.call('hgetall', key)
local last_time = tonumber(bucket[1] or now)
local tokens = tonumber(bucket[2] or capacity)

-- 2. 计算从上次到现在应该补充的令牌数
local add_tokens = (now - last_time) * rate
if add_tokens > 0 then
    tokens = math.min(tokens + add_tokens, capacity) -- 不超过桶容量
    last_time = now
end

-- 3. 判断是否有足够令牌，扣减令牌
local allow = false
if tokens >= apply then
    tokens = tokens - apply
    allow = true
end

-- 4. 保存更新后的状态
redis.call('hmset', key, 'last_time', last_time, 'tokens', tokens)
redis.call('expire', key, 3600) -- 设置过期时间，避免垃圾数据

-- 5. 返回结果：是否允许，剩余令牌数
return {allow, tokens}
```

#### 3. 服务节点调用逻辑（Java 伪代码）









```
public boolean acquireToken(String bucketKey, int rate, int capacity, int applyCount) {
    // 1. 准备参数：令牌桶key、速率、容量、当前时间戳、申请令牌数
    List<String> keys = Collections.singletonList(bucketKey);
    List<String> args = Arrays.asList(
        String.valueOf(rate),
        String.valueOf(capacity),
        String.valueOf(System.currentTimeMillis() / 1000), // 秒级时间戳
        String.valueOf(applyCount)
    );
    // 2. 执行Lua脚本
    List<Object> result = (List<Object>) redisTemplate.execute(
        new DefaultRedisScript<>(LUA_SCRIPT, List.class),
        keys,
        args.toArray()
    );
    // 3. 解析结果：result[0]是是否允许，result[1]是剩余令牌数
    return (Boolean) result.get(0);
}
```

#### 4. 令牌 “写” 到哪里？怎么分发？

- 存储位置：令牌的核心状态（`last_time`最后补充时间、`tokens`当前令牌数）**存在 Redis 的 Hash 结构**中，key 是限流标识（如 “order_api:token_bucket”）；
- 分发逻辑：无需主动 “推送” 令牌，而是**按需申请**—— 服务节点每次处理请求前，调用上述脚本向 Redis 申请令牌，Redis 原子性扣减后返回 “是否允许”；
- 令牌补充：无需单独的定时任务，每次申请令牌时，脚本会自动计算 “从上一次申请到现在应该补充的令牌数”，并更新令牌池，相当于 “懒加载” 生成令牌，避免定时任务的开销。

### 四、分布式令牌桶的优化与容错

#### 1. 性能优化：减少 Redis 请求

- 本地缓存令牌：每个节点从 Redis 批量申请令牌（如一次申请 100 个），本地缓存，用完再批量申请，减少网络开销；
- 示例：节点 A 一次从 Redis 申请 100 个令牌，本地处理 100 个请求后，再去 Redis 申请，大幅降低 Redis 调用次数。

#### 2. 高可用容错：Redis 挂了怎么办？

- 降级策略：Redis 不可用时，切换为单机令牌桶（每个节点按 “总阈值 / 节点数” 本地限流），保证服务不挂；
- Redis 集群：用 Redis 主从 + 哨兵 / 集群，避免单点故障；
- 超时处理：调用 Redis 超时，默认允许请求（或走单机限流），避免限流组件本身导致服务不可用。

#### 3. 节点扩缩容适配

- 动态调整速率：如果节点数变化，通过配置中心动态调整 Redis 令牌桶的生成速率（如节点从 10 个变 20 个，速率从 1000QPS 调为 2000）；
- 本地批量令牌过期：批量申请的令牌设置过期时间，避免节点下线后，缓存的令牌浪费。

------

### 总结

1. **分布式限流核心**：将限流状态（计数器 / 令牌）收敛到**Redis 等中心化存储**，所有节点共享状态，保证集群整体可控；
2. **分布式令牌桶关键**：令牌池存在 Redis（Hash 结构），通过 Lua 脚本原子化完成 “补令牌 + 扣令牌”，节点按需申请，无需主动推送；
3. **落地要点**：平衡一致性与性能（优先 Redis + 最终一致性），做好容错（Redis 挂了降级单机），优化网络开销（批量申请令牌）。
